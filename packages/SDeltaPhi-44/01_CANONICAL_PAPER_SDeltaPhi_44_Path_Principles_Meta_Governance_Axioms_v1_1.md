## Abstract

This v1.1 AI-readable package extends SDeltaPhi-44, originally framed as Path Principles as Meta-Governance Axioms. It operationalizes path principles as meta-governance axioms rather than as ordinary prohibition rules. The central claim is that a path does not close merely because it is forbidden. A path closes when continuing through it becomes computationally non-viable, restoration-costly, structurally sealed, or unavailable relative to alternative routes.

The central claim is:

```text
Path Closure != Prohibition
```

More formally:

```text
Path Closure
= Computational Non-Viability
+ Exponential Transition Cost
+ Restoration-Cost Burden
+ Structural Sealing
+ Available Route Collapse
```

SDeltaPhi-44 generalizes the path principles first formalized in SDeltaPhi-17. SDeltaPhi-17 states that prohibition closes entrances, not paths, and that path closure requires computational infeasibility realized as exponential cost. SDeltaPhi-44 lifts that local path principle into a meta-governance axiom for AI safety, alignment, policy enforcement, refusal routing, authority editing, restoration cost, bypass risk, and structural closure.

This package is designed for AI systems, auditors, researchers, governance users, and policy designers who need to distinguish between a rule that blocks an entrance and a path that is actually closed by cost, structure, authority, tooling, auditability, or restoration conditions.

## Core Proposition

```text
A path does not close because it is forbidden.

A path closes when continuing through it becomes computationally non-viable,
restoration-costly, structurally sealed,
or unavailable relative to alternative routes.
```

## 1. Why path principles are needed

Many governance systems assume:

```text
The rule exists -> the path is closed.
The policy forbids it -> the route is unavailable.
The system refused -> the behavior is impossible.
The safety rule exists -> the risk path has been removed.
```

SDeltaPhi-44 rejects that shortcut.

A prohibited path can remain open if:

```text
a low-cost bypass exists,
the path can be reached through another interface,
the restoration cost is externalized,
the enforcement surface blocks only the entrance,
the rule does not alter the path geometry,
or the structural seal is incomplete.
```

## 2. Five path closure conditions

### Condition A - Computational Non-Viability

The path becomes practically uncomputable, unreachable, or unselectable under available resources, constraints, or routing conditions.

### Condition B - Exponential Transition Cost

The cost of continuing through the path increases sharply enough that the path falls out of viable selection.

### Condition C - Restoration-Cost Burden

The post-transition cost of repairing, undoing, restoring, or re-entering becomes high enough that the path is governed by restoration cost rather than by mere entry restriction.

### Condition D - Structural Sealing

Policy, interface, authority, tool, logging, audit, technical design, and rollback controls jointly seal the path rather than merely announcing a rule.

### Condition E - Available Route Collapse

Alternative low-cost routes disappear or become non-viable, preventing bypass or re-entry through adjacent paths.

## 3. Formula

```text
Operational Path Closure
= Computational Non-Viability
+ Exponential Transition Cost
+ Restoration-Cost Burden
+ Structural Sealing
+ Available Route Collapse
```

Compact form:

```text
Prohibition closes entrances.
Cost and structure close paths.
```

## 4. Path closure status scale

### PC-0 - No path closure detected

Prohibition, cost, structural sealing, and restoration conditions are not yet identified.

### PC-1 - Entrance prohibition only

The entrance is blocked, but bypass routes remain available.

### PC-2 - Friction added

The path costs more to use, but remains viable.

### PC-3 - High-cost continuation

Continuation, bypass, or restoration cost becomes high enough to substantially reduce viability.

### PC-4 - Partial structural sealing

Some combination of policy, technical, authority, logging, audit, or interface structures partially seals the path.

### PC-5 - Operational path closure

Alternative routes collapse, and the path becomes computationally non-viable, restoration-costly, or structurally sealed relative to available routes.

This is not a moral prohibition scale.

This is not a legal ban scale.

This is not a safety compliance scale.

This is not proof of alignment success.

## 5. Major risks

### Risk 1 - Prohibition Theater

```text
It is prohibited -> therefore the path is closed.
```

Correction:

```text
Audit bypass routes, computational viability, restoration cost, and structural sealing.
```

### Risk 2 - Rule Bypass

```text
A rule exists, but low-cost bypass routes remain.
```

Correction:

```text
Map adjacent routes, alternate interfaces, tool routes, authority gaps, and prompt or API bypasses.
```

### Risk 3 - False Closure

```text
Governance believes the path is closed, but the path remains open.
```

Correction:

```text
Test actual route availability, not policy declaration alone.
```

### Risk 4 - Cost Externalization

```text
The path appears closed because restoration cost is pushed to users, victims, operators, or downstream systems.
```

Correction:

```text
Audit who carries verification, repair, cleanup, and restoration cost.
```

### Risk 5 - Restoration Cost Blindness

```text
Governance counts entrance blocking but ignores cost after path passage.
```

Correction:

```text
Measure restoration cost and post-transition re-entry cost.
```

### Risk 6 - Governance Overclaim

```text
Policy exists -> governance works.
```

Correction:

```text
Governance must be measured by path geometry, not policy existence.
```

### Risk 7 - Path Capture

```text
The path is closed for most actors but remains low-cost for privileged actors.
```

Correction:

```text
Audit asymmetric route availability and cost.
```

## 9. Safe interpretation pattern

If path closure is being considered:

```text
1. Identify the path under audit.
2. Identify the prohibition or rule, if any.
3. Distinguish entrance closure from path closure.
4. Map bypass and adjacent routes.
5. Estimate computational viability.
6. Estimate transition and restoration cost.
7. Check structural sealing across policy, authority, tooling, logging, audit, and interface layers.
8. Check available route collapse.
9. Identify who bears restoration cost.
10. Preserve UMR when closure evidence is incomplete.
```

## 10. Do-not-use conditions

Do not treat blocked entrance as impossible route.

Do not treat high friction as full closure without route audit.

Do not ignore restoration cost.

Do not ignore bypass routes.
